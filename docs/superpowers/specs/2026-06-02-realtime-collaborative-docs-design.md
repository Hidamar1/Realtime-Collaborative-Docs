# 多人实时编辑文档设计规格书

> 日期：2026-06-02  
> 状态：待用户审查（已根据产品、技术、市场与综合评审修订）  
> 范围：产品 + 技术混合规格  
> 方向：教育与轻项目场景优先的实时协同核心 MVP

## 1. 背景与目标

本产品面向需要高频协作的小团队，提供类似 Google Docs 的多人实时文档编辑体验。第一阶段优先聚焦 **高校实验室、课程项目组、培训机构班级小组** 等教育与轻项目协作场景，让成员可以在同一篇富文本文档中同时编辑、评论、沉淀版本，并通过文档级权限控制查看和编辑能力。

本规格经过评审后，将产品定位从「通用轻量级文档」收缩为「教育与轻项目场景优先的协作文档」。这样做的原因是：通用文档市场已有腾讯文档、飞书文档、语雀等成熟竞品，第一版如果只复刻功能清单，难以形成差异化。MVP 应先在一个具体场景中把核心协作体验做深，再扩展到更宽的团队文档场景。

第一版采用「实时共同编辑优先」策略。实时编辑、在线状态、协作者光标、自动保存和冲突处理是核心能力；评论、版本历史、权限和模板进入第一版，但保持轻量。评审建议中的离线编辑、复杂审批流、团队自定义模板、深度 IM 集成和知识库发布能力不进入 MVP。

## 2. 产品定位与差异化

### 2.1 产品定位

为高校实验室、课程项目组和培训机构班级小组提供一款实时协作文档工具，帮助团队把会议记录、课程作业、科研计划和项目报告快速转化为可共同编辑、可评论、可追溯的协作文档。

一句话定位：

> 面向教育与轻项目团队的实时协作文档，让小组成员能围绕同一份材料同步编辑、行内评论、追踪版本，并快速从模板开始协作。

### 2.2 核心差异化

第一版不把「轻量级」作为抽象卖点，而把差异化落在可感知的协作场景上：

- **教育与轻项目模板优先：** 提供实验室周报、课题计划、课程小组报告、会议纪要、项目计划等模板。
- **行内评论体验优先：** 支持选中文本评论、回复和解决，适合批注、评审和小组讨论。
- **实时协同稳定性优先：** 优先保证 2-10 人同时编辑时内容不丢失、不互相覆盖。
- **分享带动增长：** 支持公开只读预览链接，让未注册用户可以查看分享文档；编辑、评论和版本恢复仍需要登录。
- **低门槛启动：** 用户可以从模板创建文档，并立即邀请协作者进入同一份文档。

### 2.3 不作为第一版差异化的内容

以下能力可以进入长期路线图，但不作为 MVP 差异化：

- 与飞书、企业微信、钉钉的深度集成。
- 大规模知识库发布。
- AI 摘要与 AI 写作。
- 离线编辑。
- 复杂项目管理或审批流。
- 私有化部署。

## 3. 目标用户与典型场景

### 3.1 目标用户

第一版面向小团队协作场景：

- 单篇文档通常有 2-10 人同时在线。
- 用户包括教师、学生、实验室成员、培训机构老师、课程项目组成员，以及类似规模的产品、研发、设计、运营小团队。
- 编辑、评论、恢复版本和权限管理必须登录后才能执行。
- 公开只读分享链接可以允许未登录用户查看文档预览，但不能编辑或评论。

### 3.2 典型场景

- **课程小组报告：** 多名学生共同编辑报告，老师或组长通过行内评论给出修改意见。
- **实验室周报：** 成员从模板创建周报，持续补充进展、问题和下周计划。
- **课题计划：** 研究小组共同维护研究目标、任务拆分和参考资料。
- **会议纪要：** 多人同时记录会议内容，并将行动项沉淀为任务列表。
- **项目计划：** 小团队共同维护目标、任务和时间线。
- **需求文档：** 产品、设计、研发围绕同一文档协作。

## 4. MVP 范围

### 4.1 必须包含

#### 富文本实时协同编辑

- 支持段落、标题、普通列表、任务列表、加粗、斜体、链接。
- 支持多人同时编辑。
- 显示在线用户列表。
- 显示协作者光标和选区。
- 支持自动保存。

任务列表是第一版新增的轻量协作块，用于会议行动项、课程任务和项目跟进。不引入完整项目管理模块。

#### 文档级权限

- `Owner`：拥有文档，管理成员权限，删除和恢复文档。
- `Editor`：可以编辑正文、评论、恢复版本。
- `Viewer`：只读，可以查看评论和历史版本。
- `AnonymousViewer`：通过公开只读链接查看文档预览，不能查看私有成员信息，不能评论、编辑或恢复版本。

#### 分享与增长入口

- `Owner` 可以开启或关闭公开只读分享链接。
- 开启后，未登录用户可以查看文档内容预览。
- 未登录用户想要编辑、评论或加入协作时，必须注册或登录。
- 分享预览页面应突出「登录后可协作编辑」的转化入口。

#### 评论

- 支持选中文本添加评论。
- 支持评论回复。
- 支持解决评论。
- 支持侧边栏查看未解决评论。
- 不支持 @ 提及、通知中心和评论权限细分。

#### 版本历史

- 系统自动生成用户可见快照。
- 用户可以查看历史版本。
- 用户可以恢复到某个历史版本。
- 存储层可以采用「快照 + CRDT 增量日志」降低存储成本，但第一版不提供逐操作回放。
- 不支持手动命名版本。

#### 系统内置模板

第一版提供以下系统模板：

- 会议纪要。
- 项目计划。
- 需求文档。
- 周报。
- 实验室周报。
- 课程小组报告。
- 课题计划。
- 作业批注模板。

用户可以从模板创建文档。模板内容会复制为新的文档内容，普通用户不能直接编辑模板本身。

### 4.2 不进入第一版

- 大规模公开协作。
- 段落级或字段级权限。
- 表格、图片、附件、嵌入式媒体。
- AI 写作助手。
- 离线编辑。
- 移动端深度适配。
- 团队自定义模板。
- 复杂审批流。
- 完整通知系统。
- 版本对比 diff。
- 与 IM 或项目管理工具的深度集成。
- 知识库对外发布站点。

## 5. 核心用户流程

### 5.1 创建文档

用户进入文档首页后，可以新建空白文档，也可以从系统内置模板创建文档。

从模板创建文档时，系统复制模板内容，创建一篇新的用户文档。新文档创建者成为 `Owner`，文档初始内容立即保存，并自动进入编辑器页面。

### 5.2 邀请协作者

`Owner` 在文档中打开「共享/权限」面板，可以添加成员或开启公开只读分享链接。

第一版只支持文档级权限：

- `Viewer`：只能查看。
- `Editor`：可以编辑正文、评论和恢复版本。
- `Owner`：可以管理权限、删除文档、恢复版本。
- `AnonymousViewer`：只能通过公开链接查看预览，不能进入协作状态。

公开链接默认关闭。开启后，未登录用户可以查看文档内容预览，但不能编辑、评论或查看完整成员权限信息。

### 5.3 多人实时编辑

当多个 `Editor` 打开同一篇文档：

- 每个人都能编辑富文本内容。
- 编辑操作实时同步到其他在线用户。
- 每个协作者显示一个颜色标识。
- 用户能看到其他人的光标或选区位置。
- 系统自动保存编辑结果。

当 `Viewer` 打开文档：

- 可以看到实时内容变化。
- 可以看到在线协作者。
- 不能编辑正文。
- 评论和版本历史按只读权限展示。

当 `AnonymousViewer` 打开公开分享链接：

- 可以查看文档内容预览。
- 不能看到精细成员权限列表。
- 不能建立协同编辑 WebSocket 会话。
- 不能评论、编辑或恢复版本。
- 可以通过页面入口注册或登录后申请加入协作。

### 5.4 评论协作

`Editor` 或 `Owner` 选中文本后可以添加评论。评论支持创建、回复和解决。侧边栏默认展示未解决评论，已解决评论可通过筛选查看。

`Viewer` 可以查看评论，但不能新增、回复或解决评论。`AnonymousViewer` 默认不显示评论线程，避免公开预览暴露内部讨论。

### 5.5 任务列表协作

用户可以在文档正文中创建任务列表，用于记录行动项。

第一版任务列表只支持：

- 创建待办项。
- 勾选或取消勾选。
- 编辑待办文本。
- 通过实时协同同步任务状态。

第一版不支持：

- 指派负责人。
- 截止日期。
- 提醒通知。
- 外部项目管理工具同步。

### 5.6 查看与恢复版本

系统自动生成版本快照。用户可以打开版本历史面板，查看快照列表、预览历史版本，并在有权限时恢复到某个版本。

恢复历史版本视为一次新的文档变更，不删除旧历史记录。在线用户会收到恢复后的新文档状态。

### 5.7 自动保存与异常提示

编辑器持续把本地操作同步到服务端。客户端展示保存状态：

- 已保存。
- 保存中。
- 网络异常，正在重试。
- 连接断开，暂停编辑。

第一版不支持离线编辑。连接断开后，编辑器暂停编辑或切换为只读，避免产生无法同步的本地变更。恢复连接后，客户端重新拉取最新文档状态。

## 6. 系统架构

### 6.1 总体架构

系统采用三层结构：

1. 浏览器客户端。
2. 后端服务。
3. 数据存储。

浏览器客户端负责编辑器渲染、实时状态展示和用户交互。后端服务负责实时协同、权限、文档、评论、版本历史、模板和分享预览能力。数据存储负责保存文档元数据、内容状态、权限关系、评论、快照、CRDT 增量日志和模板定义。

### 6.2 客户端职责

客户端负责：

- 渲染富文本编辑器。
- 把本地编辑转换为 CRDT 更新。
- 接收远端更新并更新编辑器状态。
- 展示在线用户、光标、选区。
- 展示自动保存状态。
- 发起评论、版本、权限、模板、分享等普通 API 请求。
- 在断线时暂停编辑并提示用户。

客户端不负责：

- 决定最终文档权限。
- 绕过服务端权限判断。
- 离线保存未同步编辑。
- 向未登录公开预览用户暴露内部协作状态。

### 6.3 后端职责

后端分为实时路径和非实时路径。

实时路径负责：

- 使用 WebSocket 建立文档会话。
- 校验用户是否有文档访问权限。
- 接收 CRDT 更新。
- 合并并广播更新。
- 管理在线用户、光标、选区等 presence 信息。
- 将文档状态持久化。
- 写入 CRDT 增量日志。
- 触发自动快照条件。

非实时路径负责：

- 文档创建、读取、删除。
- 权限管理。
- 公开只读分享链接管理。
- 评论创建、回复、解决。
- 版本列表、版本预览、版本恢复。
- 模板列表、从模板创建文档。

### 6.4 数据存储职责

数据层至少需要保存：

- 文档基础信息：标题、创建者、更新时间、删除状态、分享状态。
- 文档内容状态：当前富文本结构或协同状态。
- 文档成员权限：`Owner`、`Editor`、`Viewer`。
- 评论：评论内容、作者、状态、绑定范围。
- 版本数据：用户可见快照、CRDT 增量日志、触发原因。
- 模板：模板标题、描述、初始内容。
- 审计日志：权限变更、分享开关、版本恢复、删除等关键动作。

### 6.5 推荐技术方向

本规格给出推荐技术栈，用于降低实时协同风险：

- **编辑器：** ProseMirror 或基于 ProseMirror 的成熟封装。
- **协同算法：** Yjs（CRDT）。
- **实时通信：** WebSocket，承载 Yjs 更新和 presence。
- **后端运行时：** MVP 可使用 Node.js，但 CRDT 合并、编码和压缩等计算密集任务应隔离到 `worker_threads` 或独立服务，避免阻塞主事件循环。
- **普通业务接口：** HTTP API，承载文档、权限、评论、版本、模板和分享能力。
- **数据库：** PostgreSQL + JSONB 保存业务数据、文档状态和快照。
- **活跃文档缓存：** Redis 缓存活跃文档状态和 presence 辅助数据。
- **版本存储：** 快照 + CRDT 增量日志，避免只存全量快照造成存储膨胀。

如果后续并发规模明显扩大，可以评估 Go 或 Elixir/Phoenix 承担实时连接层，但这不作为 MVP 前置条件。

### 6.6 边界原则

- 实时协同服务只处理编辑操作、CRDT 更新和 presence，不直接承担权限管理、模板管理等业务逻辑。
- 权限服务是所有读写动作的统一入口，实时连接和 HTTP API 都必须校验权限。
- 评论、版本和模板通过稳定接口接入文档，不混入编辑器核心逻辑。
- 版本恢复被视为一次新的文档变更，通过实时协同服务同步给在线用户。
- 公开分享链路只提供只读预览，不进入实时协同房间。

## 7. 数据模型与核心实体

### 7.1 `User`

表示系统用户。

核心字段：

- `id`
- `name`
- `email`
- `avatarUrl`
- `createdAt`

### 7.2 `Document`

表示文档元数据。

核心字段：

- `id`
- `title`
- `ownerId`
- `visibility`: `private | public_readonly`
- `createdAt`
- `updatedAt`
- `deletedAt`
- `lastSavedAt`

说明：

- `ownerId` 指向创建者。
- `visibility` 控制是否开启公开只读预览。
- 文档内容不直接放在 `Document` 中，避免元数据和大体积内容耦合。
- 删除采用软删除，便于后续恢复和审计。

### 7.3 `DocumentContent`

表示文档当前内容状态。

核心字段：

- `documentId`
- `contentState`
- `collaborationState`
- `version`
- `updatedAt`

说明：

- `contentState` 保存可渲染的富文本结构。
- `collaborationState` 保存 Yjs 等 CRDT 协同状态。
- `version` 表示服务端当前文档状态版本，用于保存状态和快照关联。
- 具体格式取决于最终编辑器和协同算法，但必须能表达段落、标题、普通列表、任务列表、加粗、斜体和链接。

### 7.4 `DocumentMember`

表示用户对文档的权限。

核心字段：

- `documentId`
- `userId`
- `role`: `owner | editor | viewer`
- `invitedBy`
- `createdAt`
- `updatedAt`

约束：

- 每篇文档至少有一个 `Owner`。
- `Owner` 可以变更其他成员权限。
- `Editor` 可以编辑正文、评论、恢复版本。
- `Viewer` 只读。
- 第一版不支持段落级权限。

### 7.5 `ShareLink`

表示文档公开只读分享链接。

核心字段：

- `id`
- `documentId`
- `token`
- `enabled`
- `createdBy`
- `createdAt`
- `disabledAt`

说明：

- `ShareLink` 只授予公开只读预览能力。
- 分享链接不能用于编辑、评论、恢复版本或查看内部成员权限。
- `Owner` 可以关闭链接，关闭后旧链接失效。

### 7.6 `CommentThread`

表示绑定到文档内容某个范围的一组评论。

核心字段：

- `id`
- `documentId`
- `anchor`
- `status`: `open | resolved`
- `createdBy`
- `createdAt`
- `resolvedBy`
- `resolvedAt`

`anchor` 用于描述评论绑定的文本范围。第一版需要保存足够信息，以便文档变化后尽量重新定位评论，例如：

- 起始位置。
- 结束位置。
- 被选中的原始文本片段。
- 所属块或附近上下文。

如果文本变化导致无法精确定位，评论仍保留在侧边栏，并显示「原位置可能已变化」。

### 7.7 `CommentReply`

表示评论线程中的单条回复。

核心字段：

- `id`
- `threadId`
- `body`
- `createdBy`
- `createdAt`
- `updatedAt`
- `deletedAt`

第一版不做复杂富文本评论，评论正文可以是纯文本或轻量富文本。

### 7.8 `DocumentSnapshot`

表示自动生成的用户可见历史版本。

核心字段：

- `id`
- `documentId`
- `contentState`
- `collaborationState`
- `documentVersion`
- `createdAt`
- `createdBy`
- `reason`: `time_interval | significant_change | restore | last_editor_left`

说明：

- 快照保存恢复所需的完整文档状态。
- 恢复历史版本会创建一条新的快照或变更记录，不删除旧快照。
- 第一版不支持用户可见的逐操作回放。

### 7.9 `DocumentUpdateLog`

表示 CRDT 增量日志，用于降低快照存储压力，并支持从最近快照重放到目标版本。

核心字段：

- `id`
- `documentId`
- `baseSnapshotId`
- `updatePayload`
- `documentVersion`
- `createdAt`
- `createdBy`

说明：

- `updatePayload` 保存 CRDT 增量数据。
- 增量日志是内部存储机制，不直接作为用户可见版本历史。
- 系统应定期压缩日志并生成新快照，避免无限增长。

### 7.10 `Template`

表示系统内置模板。

核心字段：

- `id`
- `title`
- `description`
- `category`
- `contentState`
- `isSystem`
- `createdAt`
- `updatedAt`

说明：

- 第一版只支持 `isSystem = true` 的系统模板。
- 从模板创建文档时，复制模板内容为新的 `DocumentContent`。

### 7.11 `PresenceSession`

表示用户在线协作状态，通常存在内存或短期存储中，不作为长期业务数据。

核心字段：

- `documentId`
- `userId`
- `connectionId`
- `cursor`
- `selection`
- `color`
- `lastSeenAt`

说明：

- 用于在线用户列表、光标、选区展示。
- 断开连接后应及时移除或过期。
- 不参与版本历史持久化。
- 不对 `AnonymousViewer` 暴露。

## 8. 实时协同与同步策略

### 8.1 协同模型

第一版推荐使用 Yjs 作为 CRDT 协同模型。

选择 CRDT 的原因：

- 多人同时编辑时能自动合并变更。
- 适合富文本结构。
- 比手写冲突处理更可靠。
- 对断线重连、乱序消息和多客户端编辑更友好。
- 后续具备支持离线编辑的技术基础，但 MVP 不启用离线编辑。

实现时应优先选择成熟 CRDT 和协同编辑生态，不从零实现协同算法。

### 8.2 同步通道

实时编辑使用 WebSocket。

连接流程：

1. 用户打开文档。
2. 客户端通过 HTTP 获取文档元数据和访问权限。
3. 客户端建立 WebSocket 连接并加入文档房间。
4. 服务端校验用户是否有 `Viewer`、`Editor` 或 `Owner` 权限。
5. 服务端发送当前文档协同状态。
6. 客户端开始接收实时内容和 presence 更新。

权限要求：

- `Viewer` 可以加入实时房间并接收文档变化。
- `Viewer` 不能发送编辑操作。
- `Editor` 和 `Owner` 可以发送编辑操作。
- `AnonymousViewer` 不能加入实时房间，只能读取公开只读预览接口。
- 服务端必须拒绝无权限用户的连接或操作。

### 8.3 编辑操作同步

当 `Editor` 修改文档：

1. 客户端将本地编辑转换为 CRDT 更新。
2. 客户端通过 WebSocket 发送更新。
3. 服务端校验用户权限。
4. 服务端合并更新到文档状态。
5. 服务端广播更新给同一文档的其他在线用户。
6. 服务端写入增量日志，并按策略持久化快照。

原则：

- 客户端可以乐观显示本地编辑。
- 服务端仍是权限和持久化的最终裁决者。
- 如果操作被拒绝，客户端必须提示并重新同步服务端状态。
- CRDT 计算和压缩任务不能长期阻塞主事件循环。

### 8.4 Presence 同步

Presence 包括：

- 在线用户。
- 用户颜色。
- 光标位置。
- 文本选区。

Presence 通过 WebSocket 广播，但不进入版本历史。

规则：

- 每个用户进入文档时分配一个稳定会话颜色。
- 光标和选区变化应节流发送，避免过度广播。
- 用户断开连接后，从在线列表移除。
- `Viewer` 可以展示 presence，但不能产生编辑操作；第一版可以只显示 `Editor` 和 `Owner` 的光标。
- `AnonymousViewer` 不参与 presence。

### 8.5 自动保存

自动保存由服务端基于协同状态持久化完成。

保存策略：

- 编辑操作实时进入协同服务内存状态。
- 服务端写入 CRDT 增量日志，降低崩溃后的数据丢失风险。
- 活跃文档状态可以缓存在 Redis 中。
- 服务端按时间间隔或变更量生成快照，并写入 PostgreSQL。
- 客户端显示保存状态：保存中、已保存、网络异常。
- 文档元数据的 `lastSavedAt` 与持久化成功时间一致。

第一版不要求每个字符输入都立即写入数据库，但要求服务端崩溃风险可控。

### 8.6 断线与重连

第一版不支持离线编辑。

断线时：

- 客户端提示连接异常。
- 编辑器进入暂停编辑或只读状态。
- 未确认的本地操作如果无法保证已提交，应要求重新同步。
- 重连后客户端拉取最新协同状态。
- 如果本地状态与服务端不一致，以服务端状态为准。

### 8.7 快照触发

版本快照由服务端自动触发。

触发条件包括：

- 固定时间间隔，例如每 10-30 分钟一次。
- 达到一定变更量。
- 用户恢复历史版本之后。
- 文档关闭或最后一个协作者离开时。

第一版只需要保证用户能看到有意义的历史版本，不需要每次微小编辑都生成用户可见快照。

## 9. 权限、评论、版本历史、模板和分享规则

### 9.1 权限规则

第一版只支持文档级权限。

| 角色 | 查看文档 | 编辑文档 | 管理成员 | 删除文档 | 评论 | 查看版本 | 恢复版本 | 开关公开预览 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| `Owner` | 是 | 是 | 是 | 是 | 是 | 是 | 是 | 是 |
| `Editor` | 是 | 是 | 否 | 否 | 是 | 是 | 是 | 否 |
| `Viewer` | 是 | 否 | 否 | 否 | 只读 | 是 | 否 | 否 |
| `AnonymousViewer` | 公开预览 | 否 | 否 | 否 | 否 | 否 | 否 | 否 |

权限校验点：

- 打开文档。
- 打开公开预览。
- 加入实时协同房间。
- 发送编辑操作。
- 创建、回复、解决评论。
- 查看、恢复版本。
- 管理成员。
- 删除文档。
- 开启或关闭分享链接。

### 9.2 评论规则

评论绑定到用户选中的文本范围。

评论线程状态：

- `open`
- `resolved`

功能规则：

- `Editor` 和 `Owner` 可以创建评论。
- `Editor` 和 `Owner` 可以回复评论。
- `Editor` 和 `Owner` 可以解决评论。
- `Viewer` 只能查看评论。
- `AnonymousViewer` 不查看评论。
- 已解决评论默认在侧边栏隐藏，但可通过筛选查看。

锚点策略：

- 评论保存原始选中文本和位置上下文。
- 文档变化后尽量重新定位锚点。
- 如果锚点失效，评论仍保留，但提示「原位置可能已变化」。

第一版不做：

- @ 提及。
- 通知。
- 重新打开已解决评论。
- 评论级权限。
- 评论内容复杂富文本。

### 9.3 版本历史规则

版本历史基于自动快照。

功能：

- 查看快照列表。
- 预览快照内容。
- 恢复某个快照。

恢复规则：

- `Owner` 和 `Editor` 可以恢复。
- `Viewer` 只能预览。
- `AnonymousViewer` 不能查看版本历史。
- 恢复版本会成为当前文档的新状态。
- 恢复动作会广播给在线协作者。
- 恢复动作会生成新的快照或历史记录，保留原历史版本。

第一版不做：

- 手动命名版本。
- 用户可见的操作级回放。
- 分支版本。
- 版本对比 diff。

### 9.4 模板规则

第一版只支持系统内置模板。

功能：

- 用户可查看模板列表。
- 用户可从模板创建文档。
- 新文档会复制模板内容。
- 创建者成为新文档 `Owner`。
- 模板本身不可被普通用户编辑。

第一版不做：

- 团队自定义模板。
- 模板权限。
- 模板分类管理后台。
- 模板版本历史。

### 9.5 分享规则

公开分享用于增长和低门槛预览，但不突破安全边界。

规则：

- 公开分享默认关闭。
- 只有 `Owner` 可以开启或关闭。
- 开启后生成只读分享链接。
- 未登录用户只能访问文档内容预览。
- 公开预览不显示内部评论、版本历史、成员权限和 presence。
- 如果用户想编辑或评论，必须注册或登录。

### 9.6 功能之间的关系

- 权限是所有功能的前置条件。
- 分享链接只授予公开只读预览能力，不等同于文档成员权限。
- 评论依赖文档内容锚点，但不改变正文。
- 任务列表是正文内容的一种块类型，通过实时协同同步。
- 版本恢复会改变正文，并通过实时协同服务同步。
- 模板只影响文档初始内容，不参与后续文档生命周期。
- Presence 只表示在线状态，不进入权限、评论或版本历史。

## 10. 安全与合规要求

### 10.1 输入与内容安全

- 富文本内容必须经过服务端允许列表校验或清理，防止 XSS。
- 评论内容必须进行转义或清理。
- 链接字段应校验协议，默认只允许 `http` 和 `https`。
- 公开预览页不能渲染未经清理的 HTML。

### 10.2 传输与存储安全

- 所有 HTTP 和 WebSocket 通信必须使用 HTTPS / WSS。
- 服务端权限校验必须覆盖 HTTP API 和 WebSocket 操作。
- 敏感配置和密钥不能写入前端或公开仓库。
- 企业版或后续版本可评估静态加密；MVP 至少保证传输加密和权限隔离。

### 10.3 审计日志

以下动作应记录审计日志：

- 文档成员权限变更。
- 开启或关闭公开分享链接。
- 删除文档。
- 恢复历史版本。
- 权限拒绝事件。
- 长时间保存失败。

## 11. 错误处理

### 11.1 网络断开

- 客户端显示「连接已断开，正在重连」。
- 编辑器暂停编辑或切换为只读。
- 重连成功后重新拉取最新文档状态。
- 第一版不尝试离线合并本地编辑。

### 11.2 权限不足

当用户尝试执行无权限操作：

- 服务端拒绝请求。
- 客户端显示明确提示，例如「你没有编辑此文档的权限」。
- 如果用户权限在会话中被降级，客户端应立即更新界面并停止编辑能力。

### 11.3 保存失败

- 客户端显示「保存失败，正在重试」。
- 服务端应保留当前内存状态并继续尝试持久化。
- 如果持久化长期失败，应阻止继续编辑或提示风险，避免用户误以为内容已安全保存。

### 11.4 评论锚点失效

- 评论仍保留在评论侧边栏。
- 评论显示「原位置可能已变化」。
- 用户仍可回复或解决该评论。

### 11.5 版本恢复冲突

当有人在线编辑时恢复历史版本：

- 恢复操作作为一次新的文档变更广播。
- 在线用户看到文档状态更新。
- 系统显示「文档已由某某恢复到历史版本」。
- 不尝试把旧版本和当前版本智能合并。

### 11.6 公开分享失效

当公开分享链接被关闭或文档删除：

- 公开预览页显示链接已失效。
- 不返回文档正文、评论、版本历史或成员信息。
- 登录用户如仍有成员权限，可以通过正常文档入口访问。

## 12. 测试与验收标准

### 12.1 单元测试

覆盖：

- 权限判断。
- 分享链接开关与访问判断。
- 文档创建逻辑。
- 模板复制逻辑。
- 任务列表块状态变更。
- 评论状态流转。
- 快照触发策略。
- 版本恢复规则。
- 富文本与评论内容清理。

### 12.2 集成测试

覆盖：

- 创建文档后，创建者自动成为 `Owner`。
- `Viewer` 不能编辑或评论。
- `Editor` 可以编辑、评论、恢复版本。
- `AnonymousViewer` 只能访问公开只读预览。
- 关闭公开分享后，旧分享链接失效。
- 从模板创建文档后，内容正确复制。
- 恢复版本后，当前内容更新且历史保留。
- 评论锚点失效时，评论仍可访问。
- 任务列表勾选状态可保存并同步。

### 12.3 实时协同测试

覆盖：

- 两个 `Editor` 同时编辑不同位置，内容都保留。
- 两个 `Editor` 同时编辑相邻位置，内容不丢失。
- 一个 `Editor` 编辑时，另一个用户实时看到变化。
- 多个用户同时修改任务列表，状态不丢失。
- `Viewer` 实时看到内容变化，但不能发送编辑操作。
- `AnonymousViewer` 不能加入实时协同房间。
- 协作者光标和在线状态正确显示。
- 用户断线后从在线列表移除。
- 重连后能同步最新状态。

### 12.4 端到端验收测试

覆盖典型完整流程：

1. 用户 A 从「课程小组报告」模板创建文档。
2. 用户 A 邀请用户 B 为 `Editor`，用户 C 为 `Viewer`。
3. A 开启公开只读分享链接。
4. 未登录用户 D 可以查看公开预览，但不能评论或编辑。
5. A 和 B 同时编辑文档。
6. A 看到 B 的光标和编辑内容。
7. C 能实时查看但不能编辑。
8. B 对选中文本添加评论。
9. A 回复并解决评论。
10. A 和 B 修改任务列表，任务状态正确同步。
11. 系统生成快照。
12. A 恢复历史版本。
13. B 和 C 都看到恢复后的文档。
14. A 关闭公开分享链接，D 再次访问时看到链接已失效。

### 12.5 MVP 验收标准

MVP 通过验收需要满足：

- 单文档 2-10 人同时在线时，编辑结果不丢失、不互相覆盖。
- 常规编辑同步延迟应低于 500 ms，至少在测试环境中可重复验证。
- 富文本基础格式可以创建、编辑、同步和保存。
- 任务列表可以创建、勾选、同步和保存。
- 在线用户、光标、选区能正确展示。
- 自动保存状态准确反映保存结果。
- 网络断开时，不会让用户继续产生不可同步编辑。
- 文档级权限能限制查看、编辑、评论、版本恢复和权限管理。
- 公开只读分享可以查看预览，但不能编辑、评论或查看内部协作信息。
- 选中文本评论可以创建、回复、解决。
- 自动快照可查看和恢复。
- 教育与轻项目模板可用于创建新文档。
- 所有核心流程有自动化测试或可重复的验收步骤。

### 12.6 非功能要求

第一版非功能要求：

- 单文档 2-10 人同时在线时保持可用。
- 常规编辑同步延迟应足够低，让协作者感知为实时。
- 文档关闭后再次打开，应恢复最近成功保存内容。
- 权限校验必须在服务端执行。
- 文档内容和权限数据不能因为客户端异常而被破坏。
- 公开预览不能泄露内部评论、版本历史、成员权限或 presence。
- 后端日志应能定位连接、保存、权限拒绝、恢复版本、分享开关等关键事件。

## 13. 增长与商业化假设

本节用于记录评审建议中的市场假设，不作为 MVP 必须实现的全部功能。

### 13.1 第一波种子用户

优先寻找以下用户验证：

- 高校实验室。
- 课程项目组。
- 培训机构班级小组。
- 轻项目协作团队。

建议目标：在 M2 阶段争取 10 个种子团队试用，收集协同编辑稳定性、模板价值和评论体验反馈。

### 13.2 免费版假设

教育场景可考虑更低门槛的免费策略：

- 教育版：50 人以下免费使用。
- 单文档实时协作人数可在技术验证后从 10 人扩展到 20 人。
- 不限文档数量，但可以限制高级模板、导出能力或去水印能力。

该策略属于商业化假设，需结合实际成本和用户反馈验证。

### 13.3 增长入口

第一版应至少内置公开只读分享能力，因为文档分享是协作产品的重要增长入口。后续可以扩展：

- 高校模板分享圈。
- 教育场景教程。
- 优质模板排行榜。
- 与 IM 或项目管理工具的导入导出。

## 15. 设计决策摘要

- MVP 选择「教育与轻项目场景优先的实时协同核心」方案，而不是先做完整文档管理套件。
- 协作规模按小团队设计，单文档 2-10 人同时在线；后续可验证 20 人单文档协作。
- 编辑器采用富文本块编辑器，第一版支持基础格式和任务列表。
- 权限采用文档级 `Owner` / `Editor` / `Viewer`，并新增公开只读 `AnonymousViewer` 预览场景。
- 评论采用选中文本评论，支持创建、回复和解决。
- 版本历史采用用户可见自动快照，存储层采用快照 + CRDT 增量日志。
- 模板采用系统内置模板，优先加入教育与轻项目模板。
- 实时协同推荐使用 ProseMirror + Yjs + WebSocket。
- 持久化推荐使用 PostgreSQL + JSONB，活跃文档状态可使用 Redis 缓存。
- 第一版不支持离线编辑，断线后暂停编辑并重新同步服务端状态。
- 第一版内置公开只读分享能力，作为增长入口，但不暴露内部评论、版本历史、成员权限或 presence。
